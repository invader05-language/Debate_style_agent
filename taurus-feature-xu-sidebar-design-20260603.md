## Problem Statement

用户希望构建一个多AI对话Agent，当用户提出需求时，系统调用多款不同的AI（MIMO、DeepSeek等），让它们以"辩论赛"模式互相反复讨论，直到形成一个完整的方案。用户根据讨论结果决定是否执行，确认后系统自动完成执行流程。系统能记住历史讨论，形成知识库，越用越聪明。

## What Makes This Cool

**AI 辩论赛模式** - 不同AI针对同一个问题互相挑战，形成对抗性思维，产生更优方案。这不是简单的协作，而是通过观点碰撞激发更好的解决方案。

**三阶段自动化**：
1. 讨论阶段 → AI们自由辩论
2. 决策阶段 → 用户审核方案
3. 执行阶段 → 一旦确认，自动执行（代码生成、测试）

**记忆进化** - 系统记住每次讨论的结论和执行结果，形成知识库，下次讨论时能参考历史经验。

## 目标模型

- **MIMO** - 小米的AI模型，国内访问方便
- **DeepSeek** - 国产高性能模型，性价比高
- 未来可扩展：Claude、GPT等

## Constraints

- 学习阶段项目，用户是主要使用者
- 基于现有框架（AutoGen/CrewAI）改造，不从零开始
- 需要支持MIMO和DeepSeek两个模型
- 用户希望1个月内能看到可用版本

### 记忆系统设计

#### 存储结构（JSON）
```json
{
  "memories": [
    {
      "id": "mem_001",
      "timestamp": "2026-06-03T10:30:00Z",
      "topic": "用户登录系统设计",
      "debate_summary": "MIMO建议JWT，DeepSeek建议Session，最终选择JWT",
      "outcome": "成功实现，用户反馈良好",
      "confidence": 0.9,
      "tags": ["认证", "安全", "JWT"],
      "lessons_learned": ["JWT更适合分布式系统", "Session需要额外存储"]
    }
  ]
}
```

#### 记忆影响机制
1. **首次使用**：无记忆，辩论基于通用知识
2. **第N次使用**：系统检索相关历史记忆，注入到辩论prompt中
3. **学习效果**：辩论中引用历史经验，避免重复错误

### 错误处理策略

| 错误类型 | 处理方式 |
|---------|---------|
| API调用失败 | 重试3次，失败后切换到备用模型 |
| 辩论死循环 | 达到最大轮次后强制终止，裁判总结 |
| 输出格式错误 | 要求AI重新输出，最多重试2次 |
| 执行失败 | 记录错误，用户决定是否重试 |

### 成本估算

| 操作 | MIMO成本 | DeepSeek成本 | 总计 |
|------|---------|-------------|------|
| 单次辩论(3轮) | ~¥0.1 | ~¥0.15 | ~¥0.25 |
| 月使用(100次) | ~¥10 | ~¥15 | ~¥25 |
| 记忆存储 | 免费 | 免费 | 免费 |

**成本控制**：使用较短的prompt，限制输出长度，避免不必要的轮次。

## 辩论协议（Debate Protocol）

### 角色定义
- **正方（Pro）**：支持某个方案，负责论证其优点
- **反方（Con）**：质疑方案，负责找出问题和风险
- **裁判（Judge）**：综合双方观点，给出最终建议

### 辩论流程
```
用户输入需求
    ↓
[轮次1] 正方提出方案 → 反方质疑 → 正方回应
    ↓
[轮次2] 反方提出替代方案 → 正方质疑 → 反方回应
    ↓
[轮次3] 裁判总结，给出结构化输出
    ↓
用户审核 → 决定是否执行
```

### 终止条件
1. 达到最大轮次（默认3轮）
2. 双方达成共识
3. 用户手动终止

### 输出格式（JSON）
```json
{
  "topic": "需求描述",
  "debate_rounds": [
    {
      "round": 1,
      "pro": {"argument": "...", "confidence": 0.8},
      "con": {"argument": "...", "confidence": 0.7}
    }
  ],
  "verdict": {
    "recommendation": "最终建议",
    "winner": "pro|con|draw",
    "confidence": 0.85,
    "action_plan": ["步骤1", "步骤2"]
  }
}
```

## Premises

1. **对抗性思维是核心价值** - 多AI辩论的价值在于不同观点碰撞产生更好方案
2. **分阶段自动化** - 讨论 → 用户决策 → 自动执行，保留用户控制权
3. **记忆进化是核心** - 系统必须能从历史中学习，不是一次性工具

## Approaches Considered

### Approach A: 最简原型
- **Summary**: 用Python写一个简单脚本，调用不同AI API，实现基本辩论循环和JSON存储记忆
- **Effort**: S (1-2周)
- **Risk**: Low
- **Pros**: 快速验证核心逻辑，学习成本低，容易调试
- **Cons**: 功能有限，扩展性差，没有Web界面
- **Reuses**: 无

### Approach B: 理想架构
- **Summary**: 从头设计完整架构，包含Web界面、向量数据库、插件系统
- **Effort**: XL (2-3个月)
- **Risk**: High
- **Pros**: 功能完整，扩展性强，用户体验好
- **Cons**: 开发周期长，学习曲线陡峭，可能过度设计
- **Reuses**: 无

### Approach C: 借力现有框架 (推荐)
- **Summary**: 基于AutoGen/CrewAI等现有框架改造，添加记忆层和执行模块
- **Effort**: M (1个月)
- **Risk**: Medium
- **Pros**: 快速实现，利用成熟框架，学习现有最佳实践
- **Cons**: 依赖第三方框架，可能需要hack框架代码
- **Reuses**: AutoGen/CrewAI框架

## Recommended Approach

**选择 Approach C: 借力现有框架**

理由：
1. 你是学习阶段，需要快速看到成果
2. AutoGen/CrewAI已经解决了多Agent协作的核心问题
3. 你只需要添加辩论逻辑、记忆层和执行模块
4. 框架代码本身就是学习资源

## v2.0 新增功能设计

> 以下为 v2.0 阶段新增的功能模块，从"辩论 Agent"升级为"Multi-AI Platform"。

### 新增功能一：AI 模型管理

用户可以在前端管理页面中添加、编辑、删除 AI 模型配置，支持 5 款预置模型和自定义模型。

**预置模型：**
| 名称 | 供应商 | 模型标识 | API 格式 |
|------|--------|----------|----------|
| MIMO V2.5 Pro | mimo | mimo-v2.5-pro | OpenAI |
| DeepSeek V4 | deepseek | deepseek-chat | OpenAI |
| Claude Sonnet 4 | anthropic | claude-sonnet-4-20250514 | Anthropic |
| GPT-4o | openai | gpt-4o | OpenAI |
| Gemini 2.5 Pro | google | gemini-2.5-pro | Gemini |

**核心设计：**
- 所有模型通过统一适配器接口调用，上层逻辑不关心具体模型
- 支持三种 API 格式：OpenAI、Anthropic、Gemini
- 支持流式输出（SSE），为实时显示做准备
- API Key 加密存储，前端不暴露完整 Key
- 提供"测试连通性"功能，验证模型配置是否正确

### 新增功能二：辩论模式增强

在原有辩论模式基础上，增强以下能力：

- **多模型选择**：用户可以从已启用的模型中自由选择正方、反方、裁判，不再固定为 MIMO 和 DeepSeek
- **可配置轮次**：辩论轮次可从 1-10 轮自由配置
- **用户中途引导**：辩论过程中用户可以插入引导语，引导辩论方向
- **暂停/恢复**：支持暂停和恢复辩论

### 新增功能三：独立思考 + 综合决策模式

这是全新的工作模式，核心理念：**让每个 AI 独立分析，再由一个"旁观者"综合判断。**

**流程设计：**
1. 用户输入问题
2. 选择参与思考的 AI（至少 2 个）
3. 选择综合裁判 AI（从剩余未参与的模型中选 1 个）
4. **阶段 1 - 独立思考**：每个被选中的 AI 独立分析同一问题，互不干扰，看不到其他 AI 的输出
5. **阶段 2 - 综合决策**：所有思考结果汇总给裁判 AI，裁判不参与思考，只负责综合，输出最终方案
6. 展示结果：每个 AI 的独立思考 + 最终综合方案

**独立思考输出结构：**
```json
{
  "thinker": "DeepSeek V4",
  "analysis": "对问题的详细分析...",
  "pros": ["优点1", "优点2"],
  "cons": ["缺点1", "缺点2"],
  "recommendation": "我的建议...",
  "confidence": 0.85,
  "key_insight": "最关键的洞察"
}
```

**综合裁判输出结构：**
```json
{
  "synthesizer": "Claude Sonnet 4",
  "summary": "综合所有分析后...",
  "consensus": ["所有AI都认同的观点"],
  "divergence": ["存在分歧的观点"],
  "final_recommendation": "最终建议...",
  "confidence": 0.9,
  "action_plan": ["步骤1", "步骤2", "步骤3"],
  "best_insight": "最值得采纳的洞察（来自哪个AI）"
}
```

**与辩论模式的区别：**
- 辩论模式是"对抗"——正方反方互相挑战，通过碰撞产生方案
- 独立思考模式是"平行"——每个 AI 独立分析，避免从众效应，综合裁判汇总所有视角

### 新增功能四：AI 模型适配层

为了支持多款不同格式的 AI 模型，需要设计统一的适配层：

### v2.0 数据库新增表

```sql
-- AI模型配置表
CREATE TABLE ai_models (
    id UUID PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    provider VARCHAR(50) NOT NULL,
    model_id VARCHAR(100) NOT NULL,
    api_url VARCHAR(500) NOT NULL,
    api_key VARCHAR(500) NOT NULL,
    api_format VARCHAR(20) DEFAULT 'openai',
    max_tokens INT DEFAULT 4096,
    temperature FLOAT DEFAULT 0.7,
    is_preset BOOLEAN DEFAULT FALSE,
    is_active BOOLEAN DEFAULT TRUE
);

-- 任务表（统一辩论和独立思考）
CREATE TABLE tasks (
    id UUID PRIMARY KEY,
    type VARCHAR(20) NOT NULL,        -- 'debate' / 'think'
    topic TEXT NOT NULL,
    status VARCHAR(20) DEFAULT 'pending',
    config JSONB,
    result JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP
);

-- 任务消息表
CREATE TABLE task_messages (
    id UUID PRIMARY KEY,
    task_id UUID REFERENCES tasks(id),
    model_id UUID REFERENCES ai_models(id),
    role VARCHAR(20),                 -- pro/con/judge/thinker/synthesizer
    round_number INT,
    content TEXT NOT NULL,
    structured JSONB,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### v2.0 技术要点

**并行调用**：独立思考模式需要同时调用多个 AI，使用 `asyncio.gather()` 实现，某个 AI 调用失败不影响其他 AI。

**流式输出**：前端实时显示 AI 的思考过程，使用 SSE（Server-Sent Events）。

**成本控制**：并行调用会增加 API 成本，需要限制并发数和输出长度。
