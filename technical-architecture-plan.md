## Architecture Decision

**分阶段实现**：先 CLI 验证核心逻辑，再升级为 Web 全栈版本。
**数据库**：一步到位使用 PostgreSQL（CLI 和 Web 共享同一个数据库）。
**框架**：AutoGen（微软出品，多Agent协作成熟）。
**实时性**：WebSocket 实时推送辩论过程。
**轮次控制**：用户可手动终止或增加轮次（默认3轮，用户可干预）。
**执行策略**：沙箱自动执行代码，自动运行测试，反馈结果。
**模型集成**：使用 LangChain LLM 封装统一接口，支持 MIMO 和 DeepSeek 动态切换。
**存储策略**：一步到位使用 PostgreSQL，CLI 和 Web 共享统一存储接口，避免代码重复。
**错误处理**：统一重试策略，API调用失败重试3次，超时后切换备用模型。
**配置管理**：使用 .env 文件管理敏感配置（API Keys、数据库连接等），python-dotenv 加载。
**测试策略**：单元测试覆盖率 85%+，集成测试 70%+，E2E 核心流程 100%。使用 pytest + pytest-asyncio + httpx 测试工具栈。
**缓存策略**：使用 Redis 缓存辩论结果和记忆查询结果，减少数据库查询压力。
**并发策略**：支持多用户并发辩论，每个辩论独立隔离，使用数据库连接池和 WebSocket 房间隔离。

### 数据库设计 (PostgreSQL)

```sql
-- 辩论表
CREATE TABLE debates (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    topic VARCHAR(500) NOT NULL,
    status VARCHAR(20) DEFAULT 'pending',  -- pending/running/completed/executed
    created_at TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP,
    verdict JSONB,
    action_plan JSONB
);

-- 消息表（每轮对话）
CREATE TABLE messages (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    debate_id UUID REFERENCES debates(id),
    round_number INTEGER,
    role VARCHAR(20),  -- pro/con/judge
    content TEXT,
    model_used VARCHAR(50),
    confidence FLOAT,
    created_at TIMESTAMP DEFAULT NOW()
);

-- 记忆表
CREATE TABLE memories (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    topic VARCHAR(500),
    debate_summary TEXT,
    outcome TEXT,
    confidence FLOAT,
    tags TEXT[],
    lessons_learned TEXT[],
    embedding VECTOR(1536),  -- pgvector 扩展，用于语义搜索
    created_at TIMESTAMP DEFAULT NOW()
);

-- 执行记录表
CREATE TABLE executions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    debate_id UUID REFERENCES debates(id),
    status VARCHAR(20),  -- pending/running/success/failed
    code_generated TEXT,
    execution_result TEXT,
    error_message TEXT,
    created_at TIMESTAMP DEFAULT NOW(),
    completed_at TIMESTAMP
);
```

#### 3. 综合引擎 (think/synthesizer.py)

综合裁判 AI 不参与思考，只负责：
1. 接收所有 AI 的独立思考结果
2. 找出共识点（所有 AI 都认同的观点）
3. 找出分歧点（存在不同意见的地方）
4. 综合分析，生成最终方案
5. 输出结构化的综合结果

### v2.0 并发与性能策略

**并行调用**：独立思考模式使用 `asyncio.gather(*tasks, return_exceptions=True)`，某个 AI 调用失败不影响其他 AI，失败的结果标记为错误继续流程。

**流式输出**：使用 SSE（Server-Sent Events）实现前端实时显示 AI 的思考过程，每个 AI 的思考流独立推送。

**错误隔离**：单个 AI 调用失败时，返回错误信息而非中断整个流程，综合裁判在汇总时会标注哪些 AI 的结果不可用。

**成本控制**：并行调用会增加 API 成本，需要：
- 限制同时调用的 AI 数量
- 限制每个 AI 的输出长度
- 使用较短的 prompt
